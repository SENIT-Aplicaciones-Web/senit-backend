# Senit Platform API

Backend API for the Senit hotel management web application.

## Stack

- ASP.NET Core Web API
- .NET 10.0
- Entity Framework Core
- MySQL
- Swagger / OpenAPI
- CORS
- Resource-based localization with `resx`

## Local database

The local connection string is configured in `appsettings.Development.json`:

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "server=localhost;port=3306;database=senit_db;user=root;password=root"
  },
  "AllowedHosts": "*"
}
```

Create the empty database before running the API:

```sql
CREATE DATABASE senit_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

The API uses `Database.EnsureCreated()` on startup, so the tables are created automatically when the project runs.

## Main tables

The persistence model is aligned with the frontend resources:

- `hotels`
- `users`
- `hotel_staff_members`
- `rooms`
- `reservations`
- `guests`
- `guest_stays`
- `consumptions`
- `payments`
- `invoices`
- `cleaning_tasks`
- `notifications`
- `subscriptions`
- `subscription_payments`

`hotel_staff_members` is the intermediate table used to represent user assignments to hotels. Removing a staff member from a hotel does not delete the user account.

## Important flows

### Sign-up

`POST /api/v1/authentication/sign-up` creates:

1. A hotel.
2. An administrator user.
3. A staff assignment in `hotel_staff_members`.
4. A default `Basic` subscription for the new hotel.

### Remove staff member from hotel

`DELETE /api/v1/hotels/{hotelId}/staff/{userId}` deactivates the assignment in `hotel_staff_members`. It does not delete the user row.

### Reservations and stays

- `reservations` stores future or confirmed room reservations.
- `guest_stays` stores active or finished stays after check-in.
- `payments` stores confirmed payments.
- `invoices` stores issued receipts.
- Creating an invoice for a stay payment finalizes the stay, sets `actual_end_at`, changes the room to `cleaning`, and creates a pending cleaning task.
- `cleaning_tasks` stores room cleaning records generated by operational flows.

## Frontend local connection

The frontend should use:

```env
VITE_API_BASE_URL=http://localhost:5026/api/v1
```

or the HTTPS port shown by Rider.


## Final validation rules added

- Rooms only accept the frontend room types: `Standard`, `Deluxe`, and `Suite`.
- Room status can only be assigned manually as `available`, `cleaning`, or `maintenance`.
- `occupied` is reserved for internal check-in flow and is set by the backend when a guest stay is created.
- Reservations and guest stays can start at any minute, but their duration must be a whole number of hours. Partial-hour durations are rejected.
- Reservation overlap and active-stay conflicts return `409 Conflict` and are documented in Swagger.

## Deployment notes

A `Dockerfile` is included for cloud deployment. Configure the production database using an environment variable instead of editing source files:

```txt
ConnectionStrings__DefaultConnection=server=<host>;port=3306;database=senit_db;user=<user>;password=<password>
```

For simple academic deployment, keep `EnsureCreated()` enabled and start with an empty MySQL database. If the model changes after data exists, recreate the database or move to migrations.

The API listens on port `8080` inside the Docker container.
